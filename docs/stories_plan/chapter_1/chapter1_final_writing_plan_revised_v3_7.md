## 1. Writer Agent 核心任務

第 1 章要讓玩家第一次親手體驗全篇核心主題：

> **真實資料也可能被整理成錯誤故事。  
> 一個普通人不需要被偽造證據，就可能被 summary 排成犯人。**

但這一章不是主線爆炸章。不要急著解釋青葉舊案、A-90、第零證人、雨宮身份或 KAGAMI 的完整三層規則。

本章必須完成四件事：

1. 教玩家基本玩法：現場調查、證物欄、證詞矛盾、KAGAMI「摘要」與「本機順序／核准片段」的差別。
2. 完成一個獨立成立、邏輯乾淨、可回推的小型咖啡館殺人案。
3. 讓三宅蒼太、北見修一、神谷澪、早坂茜等早期角色留下可記憶的性格與日常面。
4. 留下主線陰影，但不提前攤牌。

最重要的寫作原則：

> **玩家破案時，不能靠猜。  
> 玩家感到不安時，可以靠猜。**

換句話說：案件骨架要講清楚；主線意義要留白。

### 2.5.3 咖啡線索功能

咖啡機 / 第二杯咖啡的功能以 V3.7 為 canon：

- 它**不單獨鑑定死亡分鐘**。
- 它**不獨立證明死亡更早**。
- 它只證明：增田當晚等過第二人，且這場會面發生得比摘要故事更早。
- 死亡更早必須由手機通知、舊掛鐘撞擊痕、法醫初步範圍、雨水乾燥圖與門鎖本機順序共同支撐。

### 2.5.4 閉店流程時間鎖

本章所有文件統一成：

| 時間 | Canon |
|---|---|

### 2.5.5 Maintenance Panel 保全鏈

店長手機截圖只作 **lead**，不是審查會 exhibit。

正式程序固定如下：

1. 店長先出示手機截圖，指出維護頁與摘要時間線不一致。
2. 黑瀨不直接收她手機截圖作證據，而是讓鑑識員現場重新拍攝本機頁面。
3. 鑑識紀錄需包含：門鎖設備 ID、機身編號、頁面版本號、當前畫面時間、`Event-1841～1844` 的本機順序。
4. 這份程序固定紀錄才可在審查會中作為正式材料。

寫作重點：店長截圖是「相馬發現不對的入口」；黑瀨 / 鑑識固定後，才變成可被審查會討論的材料。

### 2.5.6 KAGAMI Summary Narrative Anchor Rule

這條規則不是第 1 章臨時補丁，而是全篇 KAGAMI summary/report layer 的系列通則。

> 摘要層會傾向選取 **human-visible / credential-linked / camera-correlated** 的事件作為敘事時間錨。

第 1 章裡，三宅的事件最好看：員工憑證、監視器、打卡紀錄都對得上，所以被摘要排成主要 foreground timestamp。北見的外包憑證雖然更早，但身份未核、後門無正面鏡頭，因此被降成低關聯 background / queued item。

角色台詞只需口語化。玩家只需要聽見一個版本，不要反覆使用 Narrative Anchor、foreground timestamp、queued item 等術語。

> 「摘要抓到的，不一定是第一個人。  
> 它抓的是最好對得上的那個。」

審查會口語示例：

> 神谷：「門、鏡頭、員工憑證，全都對得上三宅。」  
> 相馬：「所以摘要把那一筆寫得最像真相。」

### 2.5.7 `K.` 線索平衡

`K.` 要保留，否則北見會像終盤新人物；但它不能太直白到警方像漏看 obvious clue。

固定如下：

- 第二杯咖啡上的 `K.` 是店內杯套 / 備註代碼，不是完整姓名。
- Beat 4 三宅只說「增田先生好像在等外包那邊的人」，不要太早說「北……」。
- 外包名單中可以有多個 K 開頭或代號相近的承包商。
- Beat 8～9 才把 `K.`、外包 queued item、`External Maintenance Credential`、臨時維護工單對上北見修一。

### 2.5.8 致命打擊聲音同步

聲音遮蔽不是「完全沒人聽到」，而是「有人聽到了，但把它誤認成店內設備聲」。

22:58:45 北見擊打增田的瞬間，前場主要遮蔽源只保留兩個：

1. 洗杯機 / espresso machine backflush 排氣聲。
2. 收銀抽屜彈開、硬幣盤碰撞聲。

背景氛圍可有雨打後巷鐵門、三宅在休息區通話、L 型轉角與半掩防火門造成的聲音衰減，但不要讓角色逐項解釋所有聲音條件。

店長後來可回憶：

> 「那一下我聽見了。  
> 我以為是那台機器 backflush 的震動。」

這不進證物欄，只服務現場可信度。

## 7. 術語與 UI 統一規則

第 1 章是 tutorial，不要讓玩家同時記住太多近似名詞。

### 7.1 角色口語只用三層

| 角色口語 | 正式概念 | 玩家理解 |
|---|---|---|
| 摘要 | KAGAMI summary / 警方先看的報表 | 整理後故事，不是現場本身 |
| 本機順序 | maintenance panel / local event sequence | 門鎖本機記錄的事件先後，不含精確秒數 |
| 核准片段 | 審查會核准的有限原始片段 / narrow extraction | 被程序允許查看的小範圍資料，不是完整 raw export |

禁止角色在第 1 章反覆講：`raw data`、`sync line`、`local event`、`maintenance panel`、`narrow extraction`、`merge layer`。這些可以出現在 UI 小字或證物說明中，但台詞要口語化。

### 7.2 UI 視覺語法

| 類型 | 視覺語法 |
|---|---|

## 8. 案發地點、赴約理由與殺人方式定案

### 8.1 增田為何會在後場

增田不是誤入倉庫。

他曾因「雨鐘」咖啡館智慧門鎖試點資料，到店裡做過幾次資料確認。店長認得他，也知道他與外包門鎖資料審查有關。

案發當晚，增田用「確認門鎖維護紀錄」作為表面理由，在打烊後短暫使用後場小辦公角落。

真正目的：

- 他約北見私下談資料盜賣。
- 他想要求北見交代資料流向。
- 他刻意避開 KAGAMI 辦公室與公司攝影機。
- 他帶著 USB，不是毫無準備，而是以為這次能補上檢舉鏈最後一塊。

### 8.2 小型先行保險 + 交換型會面

為避免增田像「為了讓案件發生才犯蠢」，固定如下：

- 增田已把一份普通檢舉草稿存進公司內部監察信箱草稿區，但尚未送出。
- 那份草稿只能證明北見有異常接觸紀錄，還不足以證明資料流向。
- 北見聲稱願意交出「買家鏈的一段」，條件是增田交還或刪除某個檔案。
- 增田因此帶著 USB 赴約。
- 增田不是準備犧牲，也不是毫無保險。他只是以為能補上最後一塊證據。

注意：這不是第 7 章紙本 dead-man switch。第 1 章不能做成 miniature dead-man switch。

### 8.2.1 增田低調自保痕跡（V3.7）

增田知道自己在碰危險資料，所以要留下一個很普通、很日常的痕跡。但它不能變成第 7 章等級的 dead-man switch。

固定如下：

- 增田離開公司前，打開手機行事曆，又關掉。
- 最後他只在未送出的備忘裡留下一行：`22:50 雨鐘 / 校驗値確認 / K`。
- 這行字沒有寄給任何人，也不會自動公開。
- 它前期只作死者人物筆記；中後段才轉為「增田等待第二人」的輔助材料。
- 不要讓它變成「北見就是兇手」的直線證據。

功能：增田不是毫無保險、不是為了案發才犯險。他只是做了一個真實成年人會做的低調自保，並錯估了北見會不會越線。

### 8.3 維護窗與作案時點的因果

採用 **「增田選雨鐘，北見順勢利用」**。

- 雨鐘是增田選的，因為這裡是他熟悉的門鎖試點現場，店長認得他，且能避開公司攝影機。
- 北見收到約談地點後，知道雨鐘每晚打烊前後有門鎖維護窗口。
- 北見不是一開始就計畫完美謀殺，而是在衝突後利用自己熟悉的維護規則，把三宅正常進出排進殺人時間。
- 這讓北見仍是膽怯、利己、實用主義的小人物，不變成大反派型 mastermind。

### 8.4 北見當夜即時壓力

北見不能只是抽象地「害怕被檢舉」。案發當夜要有迫近壓力，但這個壓力不能來自主線黑幕直接威脅。

固定如下：

- 增田在談判中明說：普通檢舉草稿已存進內部監察信箱草稿區。
- 北見知道自己的外包維護帳號隔天要進入合約審核／權限回收流程。
- 一旦增田送出檢舉，北見的異常存取紀錄與臨時維護憑證會被重新審計。
- 北見恐慌的來源是制度壓力與自保，不是鏡原直接命令，也不是 A90 主線威脅。

寫作效果：北見不是高階 mastermind，也不是被黑幕操控的棋子。他是一個發現「明天之前事情就會進入制度」後，為了自保而失控殺人的小人物。

## 10. 本機順序與核准片段如何合法進場

本章不讓相馬直接看到完整 raw export。

### 10.1 准入規則：narrow extraction 標準

早坂在 Beat 2 或 Beat 6 前置說明：

> 「如果辯方能提出至少兩項獨立矛盾，而且矛盾都直接指向同一裝置的時序可信度，審查會可以核准 narrow extraction。不是打開全部 raw，只能調出該裝置、該時間段、該爭議欄位。」

本章符合准入條件的矛盾：

| 矛盾 | 對應裝置 / 時序 |
|---|---|
| 死者手機通知與掉落時間早於摘要死亡時間 | 死亡時序 |
| 咖啡機最後出杯與保溫狀態不支持 23:08 後才衝突 | 會面時序 |
| 地板雨水乾燥程度顯示更早進出 | 後門進出時序 |
| 店家本機順序顯示外包憑證事件早於三宅事件 | 後場門鎖時序 |

### 10.2 第一階段：店家本機順序 lead + 程序固定

取得時機：Beat 8 反轉調查。

第一眼看見的是店長手機裡的門鎖維護面板截圖。這不是完整 raw export，也不是正式 exhibit，只是店家／物管可查看的本機維護頁。

店長截圖畫面內容：

- `Event-1841`：Maintenance Mode ON
- `Event-1842`：External Maintenance Credential / back door open
- `Event-1843`：Staff Credential / rear corridor open
- `Event-1844`：Maintenance Sync Completed

限制：

- 沒有精確秒數。
- 沒有完整使用者姓名。
- 只有事件順序、憑證類型、門磁開關狀態。

#### 保全鏈處理

不要讓店長手機截圖直接打進審查會。

固定程序：

1. 店長手機截圖讓相馬意識到摘要時間線有問題。
2. 黑瀨沒有直接接收店長手機，而是讓鑑識員現場重新固定本機頁面。
3. 鑑識固定內容包括：門鎖設備 ID、門鎖機身編號、頁面版本號、當前畫面時間、`Event-1841～1844` 的本機順序。
4. 店長截圖降為 lead；鑑識固定紀錄才是可被審查會討論的程序材料。

可用短場景：

> 黑瀨沒接店長的手機。  
> 「你這張先留著，當參照。」  
> 他讓鑑識員對著本機頁面拍下設備 ID、頁面版本和事件順序。  
> 「真正要進審查會的，不是你手機裡那張，是我現在看到的這一頁。」

玩家第一次意識到：門鎖本機不是在說「23:07:50 三宅進門」，它只說某幾件事先後發生。


### 10.3 第二階段：先核 metadata，再核准片段

取得時機：Beat 10 最終審查會中段。

審查會不要一口氣放出核准片段。要有阻力。

流程：

1. 神谷先只允許調出後場門鎖該時段的 metadata：設備 ID、同步完成時間、維護模式標記。
2. 早坂用手機通知、咖啡機最後出杯、雨水乾燥圖與本機順序，證明 metadata 與摘要時間線無法同時成立。
3. 審查會才核准後場門鎖 22:50～23:10 的有限片段。

核准片段範圍：

- 後場門鎖。
- 22:50～23:10。
- 事件序號。
- 憑證類型。
- 同步時間。
- 保全鏈標記。

不准調：

- 完整店內所有門鎖 raw。
- 全部 KAGAMI 資料。
- 北見外包資料鏈完整名單。
- A-90 相關檔案。

### 10.4 Summary foreground timestamp rule

V3.6 必須把這條規則當成全篇 summary/report layer 通則，而不是第 1 章專用補丁；否則玩家會問：「既然 `Event-1842` 和 `Event-1843` 都在同一批 queued events 裡，為什麼摘要偏偏把三宅那筆壓成主要案發時間？」

固定規則：

> 摘要層會傾向選取 human-visible、credential-linked、camera-correlated 的事件作為 narrative anchor。
>
> 維護模式下，門鎖本機只保存 queued events 的順序。Sync completed 後，摘要層會嘗試選出一筆最容易與外部資料對上的 foreground timestamp，作為敘事時間錨。

在本案中，摘要優先選中三宅事件，因為它同時符合：

- `Staff Credential`。
- 前場 / 後場監視器可見三宅靠近後場。
- 打卡紀錄證明三宅當晚在班。
- 它是 sync completed 前最容易與「可見人物」對上的 foreground access。

相反，北見的 `External Maintenance Credential` 被降權，因為：

- 身份未在第一輪摘要中完整核驗。
- 後門沒有正面 CCTV。
- 它被歸入外包維護 queued item，而不是「店內可見員工行動」。

角色台詞要簡化成：

> 「系統不是在找第一個進門的人。它是在找最好對上監視器和員工憑證的那一筆。」

不要把這段寫成技術課。玩家只需要懂：**三宅不是唯一事件；他只是被摘要選成最容易說成事件時間的那筆。**

### 10.5 店方知識邊界

店長不能變成懂系統的工程師，也不能讓所有人看起來突然變笨。固定如下：

- 店長平常只用 maintenance panel 確認門有沒有卡住、維護模式有沒有關掉。
- 她不知道這頁不能拿來當精確案發秒數。
- 警方初期拿到並沿用的是 KAGAMI 摘要匯出，不是店方操作頁。
- 店方本機頁只能成為「質疑摘要」的入口，不能單獨定案。

可用短台詞：

> 店長：「我平常只看這頁確認門有沒有卡住、維護模式有沒有關掉。我不知道它不能拿來判幾點幾分有人進門。」
>
> 神谷：「店方維護畫面可以成為質疑摘要的材料。但它不是完整 raw，也不能單獨推翻整條時間線。」

## 11. 證物功能分工

不要讓所有 clue 都只證明「死亡更早」。每個證物回答不同問題。

| 線索 | 不要讓它證明 | 應該讓它證明 |
|---|---|---|
| 咖啡機最後出杯 / 餘溫 | 不要單獨證明死亡時間 | 增田等的「第二杯咖啡的人」更早到場或至少被預期到場 |
| 第二杯咖啡備註 `K.` | 不要直接定北見有罪 | 北見存在早於後段正式登場；增田等的人不是三宅 |
| 死者手機通知 | 不要只是另一個時間錨 | 衝突最晚在 22:58 左右已發生 |
| 舊掛鐘照片 | 不要獨立定錘 | 與手機通知、撞擊痕互相咬合，證明衝突瞬間 |
| 地板雨水乾燥圖 | 不要泛泛證明死亡更早 | 證明有人在三宅之前從後門進入並走向內側倉庫 |
| 濕傘套 | 不要單獨定罪 | 證明真正濕傘使用者走承包商動線，不是三宅的店內動線 |
| `External Maintenance Credential` | 不要當作「北見親口承認」 | 將更早進入者收束到外包維護權限 |
| 臨時維護工單 | 不要單獨證明殺人 | 將外包維護權限對到北見 |
| 後場平面圖 / L 型轉角 | 不要只是空間說明 | 證明三宅 23:06 到前段走廊時不會看見內側倉庫屍體 |
| 本機順序 | 不要當完整 raw | 證明門鎖只保存順序，摘要補上的時間不是事件時間 |

## 25. Writer Agent 實作提醒

### 語氣

- 適合 YA／廣義大眾玩家，但不要太幼。
- 技術要簡潔，不要堆術語。
- 重點放在現場、人物、矛盾、證物順序。
- 日常場景要短，但要有記憶點。
- 日常裡可以加入兼職壓力、外包不安、程序羞恥感，讓人物更接近成年人世界。

### 節奏

| 區段 | 目標時長 |
|---|---:|
| 開場 + 委託 | 20～25 分鐘 |
| 第一次現場調查 | 30～35 分鐘 |
| 三宅詢問 + 第一輪審查會 | 35～40 分鐘 |
| 喘息 + 反轉調查 | 35～45 分鐘 |
| 北見調查 + 最終審查會 | 45～55 分鐘 |
| 結案 + 章末鉤子 | 10～15 分鐘 |

### 主題台詞限制

本章明確主題句最多兩次：

1. 前半一次：相馬偏信摘要。
2. 結尾一次：相馬修正自己對摘要的理解。

其他地方不要讓角色直接說「證據被排列」「摘要不是現場」等同義句，改用水痕、收據、舊掛鐘、咖啡機、L 型後場與門鎖畫面表現。

最重要的寫作原則：

> **不要讓角色說主題。  
> 讓玩家透過錯誤摘要、濕掉的傘套、壞掉的咖啡機、舊掛鐘、三宅的飯糰袋，自己感覺到主題。**

本章不是在說明「證據會被排列」。  
本章是讓玩家親手拆掉一個被排列好的故事。
