# Evidence Carrier Assignment Design

## Problem

The layout editor still treats evidence as if it always belongs to a physical
hotspot. That is too narrow for Chapter 1 investigation scenes:

- Some evidence comes from a person, such as `法醫初步死亡範圍` from 黑瀨.
- Some evidence needs a fallback standalone hidden hotspot when no existing
  object or person fits.
- Some authored hotspots, such as `後場入口`, are navigation or spatial context
  and should not be deleted or treated as evidence carriers.
- Reassigning evidence currently removes or adds reveal targets, but it does not
  have enough target typing to distinguish authored hotspots, generated
  standalone hotspots, and person-topic evidence.

The source of truth must remain authored Markdown. Generated JSON is a compiled
view and should not be edited directly.

## Decisions

Evidence assignment will use typed carriers:

```ts
type EvidenceCarrier =
  | { kind: "hotspot"; sublocationId: string; hotspotId: string }
  | { kind: "topic"; sublocationId: string; characterId: string; topicId: string }
  | { kind: "standalone_hotspot"; sublocationId: string };
```

Person-sourced evidence is collected through a character topic, not by clicking a
character directly. The editor may display this as `人物 / 話題`, but the authored
Markdown reveal belongs on the topic.

Standalone evidence hotspots are created only when no authored hotspot or topic
fits. They use an ID convention:

```text
evidence_source_<evidence_id>
```

Only hotspots with that ID shape are considered editor-generated and eligible for
automatic deletion after their final evidence reveal is reassigned away.

## Editor Behavior

The evidence assignment panel is scoped to the currently selected sublocation.
For each evidence item whose `Source Sublocation` matches the selected
sublocation, the editor offers:

- existing evidence-capable hotspots in that sublocation;
- character topics in that sublocation;
- a `Create standalone hotspot` option.

Existing hotspot targets must already have `Evidence Source` metadata, because
the editor assignment patcher does not collect new source metadata for arbitrary
authored hotspots. Standalone generated hotspots default to:

```md
- **Evidence Source:** hidden
- **Scene Source Prompt:** Hidden local evidence source generated by the layout editor; the collected evidence is not visibly readable in the background.
```

The editor should label targets by type so the user can distinguish physical
hotspots from person-topic carriers.

## Markdown Patch Rules

When assigning evidence:

1. Remove the `evidence:<id>` reveal from any existing hotspot or topic in the
   scene.
2. Add the reveal to the selected hotspot or topic, preserving other reveal
   targets.
3. If the selected target is `standalone_hotspot`, create a new hotspot block in
   the evidence source sublocation with ID `evidence_source_<evidence_id>`.
4. If the old carrier is a generated standalone hotspot and has no remaining
   evidence reveal targets, remove that hotspot block.
5. Do not remove authored hotspots such as navigation, spatial, texture, or
   non-evidence triggers, even if they no longer reveal evidence.
6. Do not delete character topics automatically. Topics are authored dialogue
   structure, not generated evidence containers.

After a successful Markdown write, the editor patches in-memory `editorState.scene`
and layout sidecar state to match. This keeps the visible editor in sync until
the scene is recompiled. The generated JSON remains stale until
`bun run scenes:compile` or an editor compile/reload command runs.

## Validation And Sync

The compiler already enforces same-sublocation evidence reveals. The editor must
honor the same rule for both hotspot carriers and topic carriers:

- A hotspot target is valid only when its parent sublocation matches the
  evidence `Source Sublocation`.
- A topic target is valid only when its character lives in the evidence
  `Source Sublocation`.
- A generated standalone hotspot is created only inside the evidence
  `Source Sublocation`.

The editor writes Markdown first. If the write succeeds and the user has not
changed scenes, it patches in-memory JSON. If the write fails, no in-memory patch
is applied. Assignment writes remain serialized so rapid changes cannot let an
older write overwrite a newer assignment.
